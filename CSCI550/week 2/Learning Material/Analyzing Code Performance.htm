1.1 Architectural Styles and Patterns

Architectural styles and patterns provide a high-level framework for organizing system components and interactions, enabling the development of maintainable and scalable software systems. These patterns define how software systems should be structured to achieve desired qualities such as performance, reliability, flexibility, and adaptability. By understanding these architectural styles, developers can build systems that are more resilient, efficient, and easier to maintain over time.

Key architectural styles include:

  • Layered Architecture: This pattern divides an application into multiple distinct layers, each responsible for a specific set of functionality. This promotes the separation of concerns and modularity. The layers typically include:
    • Presentation Layer: Handles user interface and browser communication logic
    • Business Layer: Executes necessary business logic related to the application’s functionality
    • Data Layer: Manages application data storage
    • Persistence Layer: Encapsulates long-term data storage and retrieval operations

The layered architecture divides an application into distinct layers, each handling specific responsibilities. Here’s a practical implementation: 


# Data Layer
class UserRepository: 
    def __init__(self): 
        self.users = {}  # Simulating a database 
         
    def get_user(self, user_id): 
        return self.users.get(user_id) 
     
    def save_user(self, user_id, user_data): 
        self.users[user_id] = user_data
      # Business Layer 
class UserService: 
    def __init__(self, repository): 
        self.repository = repository 
     
    def get_user_details(self, user_id): 
        user = self.repository.get_user(user_id) 
        if not user: 
            raise ValueError("User not found") 
        return user 
     
    def create_user(self, user_id, name, email): 
        user_data = {"name": name, "email": email} 
        self.repository.save_user(user_id, user_data) 
      # Presentation Layer 
class UserController: 
    def __init__(self, user_service): 
        self.user_service = user_service 
     
    def get_user(self, user_id): 
        try: 
            user = self.user_service.get_user_details(user_id) 
            return {"status": "success", "data": user} 
        except ValueError as e: 
            return {"status": "error", "message": str(e)}

This implementation demonstrates clear separation of concerns: - The Repository layer handles data persistence - The Service layer contains business logic - The Controller layer manages user interactions and request handling

The layered architecture promotes modularity and makes the system easier to maintain and test. Each layer can be modified or replaced independently without affecting other layers.

  • Microservices Architecture: This pattern decomposes an application into a collection of loosely coupled services, each responsible for a specific business function (Hooda et al., 2023). The microservices architecture promotes scalability, flexibility, and fault isolation. The services are typically independent of one another and communicate using lightweight mechanisms, such as REST APIs or messaging queues.
    • Service Independence: Each microservice focuses on a specific business capability and can be developed, deployed, and scaled independently.
    • Fault Isolation: Failure in one microservice does not necessarily affect others, improving overall system resilience.
    • Scalability: Microservices can be scaled based on individual service needs, making it suitable for large-scale applications where different components have varying performance requirements.

Microservices architecture is ideal for complex applications requiring flexibility, scalability, and the ability to support continuous deployment by allowing teams to work independently on different services.

This example shows how microservices are independently deployable, communicate via HTTP/REST, have their own data storage, and can be developed and scaled independently.

  • Event-Driven Architecture: This pattern uses events to facilitate communication between loosely coupled components (Tosi, 2023; Misra et al., 2024). It is effective for applications that need to respond in real-time to changes in the environment.
    • Producers and Consumers: Components in an event-driven architecture are categorized as producers (which generate events) and consumers (which respond to those events).
    • Asynchronous Communication: Events are communicated asynchronously, enabling real-time responsiveness and decoupling of components.
    • High Availability: Event-driven systems are often used in applications that require high availability and responsiveness, such as financial trading systems, e-commerce platforms, and IoT applications.

By using events to trigger actions, this architecture is well-suited for handling large volumes of data while maintaining efficient processing, without creating bottlenecks in the system.

Here’s an example:


    from abc import ABC, abstractmethod 
    from typing import Dict, List 
    import asyncio 
     
    # Event definitions 
    class Event: 
        def __init__(self, event_type: str, data: Dict): 
            self.type = event_type 
            self.data = data 
     
    # Abstract base class for event handlers 
    class EventHandler(ABC): 
        @abstractmethod 
        async def handle(self, event: Event): 
            pass 
     
    # Example event handlers 
    class OrderCreatedHandler(EventHandler): 
        async def handle(self, event: Event): 
            print(f"Processing new order: {event.data}") 
            # Simulate processing time 
            await asyncio.sleep(1) 
            print(f"Order {event.data['order_id']} processed") 
     
    class InventoryUpdateHandler(EventHandler): 
        async def handle(self, event: Event): 
            print(f"Updating inventory for order: {event.data}") 
            await asyncio.sleep(0.5) 
            print(f"Inventory updated for order {event.data['order_id']}") 
     
    # Event bus implementation 
    class EventBus: 
        def __init__(self): 
            self.handlers: Dict[str, List[EventHandler]] = {} 
     
        def register(self, event_type: str, handler: EventHandler): 
            if event_type not in self.handlers: 
                self.handlers[event_type] = [] 
            self.handlers[event_type].append(handler) 
     
        async def publish(self, event: Event): 
            if event.type in self.handlers: 
                await asyncio.gather( 
                    *[handler.handle(event) for handler in self.handlers[event.type]] 
                ) 
     
    # Usage example 
    async def main(): 
        # Create event bus and register handlers 
        event_bus = EventBus() 
        event_bus.register("order_created", OrderCreatedHandler()) 
        event_bus.register("order_created", InventoryUpdateHandler()) 
     
        # Publish an event 
        order_event = Event( 
            "order_created", 
            {"order_id": 123, "product": "Book", "quantity": 2} 
        ) 
        await event_bus.publish(order_event) 
     
    # Run the example 
      asyncio.run(main())

This example demonstrates an event-driven architecture with async/await patterns in Python. It includes:

An Event class to represent events in the system, an abstract EventHandler class that defines the interface for event handlers, concrete handler implementations for different business processes, an EventBus that manages event publication, and handler registration.

  • Client-Server Architecture: This pattern separates the system into two main components: clients and servers (Hooda et al., 2023). Clients are responsible for making requests, while servers provide the requested services.
    • Client Component: Manages user interactions, including user input, and sends requests to the server for processing.
    • Server Component: Handles business logic, processes client requests, and manages data storage.
    • Centralized Control: The server component manages data and provides services, allowing for centralized data control and easy maintenance.

The client-server architecture is widely used for web applications and distribution systems (Sabharwal, Rathore, & Agrawal, 2021). It provides a clear separation between user interface and backend processing, allowing for scalability and efficient data management, and is particularly effective for systems that need to ensure a consistent user experience across different client devices.

Here is a comprehensive example using Python that demonstrates both synchronous HTTP and WebSocket-based real-time communication: 


    # Server-side implementation 
    from flask import Flask, jsonify, request 
    from flask_socketio import SocketIO, emit 
    import threading 
    import time 
     
    app = Flask(__name__) 
    socketio = SocketIO(app, cors_allowed_origins="*") 
     
    # Simulated database 
    users_db = {} 
    active_connections = {} 
     
    # REST API endpoints for traditional client-server communication 
    @app.route('/api/users', methods=['POST']) 
    def create_user(): 
        """Handle user creation requests from clients""" 
        data = request.get_json() 
        if not data or 'username' not in data: 
            return jsonify({'error': 'Username is required'}), 400 
     
        username = data['username'] 
        if username in users_db: 
            return jsonify({'error': 'Username already exists'}), 409 
     
        users_db[username] = { 
            'username': username, 
            'created_at': time.strftime('%Y-%m-%d %H:%M:%S'), 
            'status': 'active' 
        } 
     
        return jsonify(users_db[username]), 201 
     
    @app.route('/api/users/<username>', methods=['GET']) 
    def get_user(username): 
        """Handle requests for user information""" 
        if username not in users_db: 
            return jsonify({'error': 'User not found'}), 404 
        return jsonify(users_db[username]) 
     
    # WebSocket handlers for real-time communication 
    @socketio.on('connect') 
    def handle_connect(): 
        """Handle new client connections""" 
        client_id = request.sid 
        active_connections[client_id] = { 
            'connected_at': time.strftime('%Y-%m-%d %H:%M:%S'), 
            'last_activity': time.time() 
        } 
        emit('connection_success', {'message': 'Connected to server'}) 
     
    @socketio.on('disconnect') 
    def handle_disconnect(): 
        """Handle client disconnections""" 
        client_id = request.sid 
        if client_id in active_connections: 
            del active_connections[client_id] 
     
    @socketio.on('message') 
    def handle_message(data): 
        """Handle real-time messages from clients""" 
        client_id = request.sid 
        if client_id in active_connections: 
            active_connections[client_id]['last_activity'] = time.time() 
            # Broadcast message to all connected clients except sender 
            emit('new_message', { 
                'sender': client_id, 
                'message': data['message'], 
                'timestamp': time.strftime('%Y-%m-%d %H:%M:%S') 
            }, broadcast=True, include_self=False) 
     
    # Background task to monitor client connections 
    def monitor_connections(): 
        """Monitor and clean up inactive connections""" 
        while True: 
            current_time = time.time() 
            inactive_clients = [ 
                client_id for client_id, data in active_connections.items() 
                if current_time - data['last_activity'] > 300  # 5 minutes timeout 
            ] 
     
            for client_id in inactive_clients: 
                del active_connections[client_id] 
                socketio.emit('disconnect', room=client_id) 
     
            time.sleep(60)  # Check every minute 
     
    # Client-side implementation (example using requests and socketio-client) 
    import requests 
    from socketio import Client 
    import json 
     
    class ChatClient: 
        def __init__(self, server_url): 
            """Initialize chat client with server URL""" 
            self.server_url = server_url 
            self.sio = Client() 
            self.setup_socket_handlers() 
     
        def setup_socket_handlers(self): 
            """Set up WebSocket event handlers""" 
            @self.sio.on('connection_success') 
            def on_connect(data): 
                print(f"Connected to server: {data['message']}") 
     
            @self.sio.on('new_message') 
            def on_message(data): 
                print(f"New message from {data['sender']}: {data['message']}") 
     
            @self.sio.on('disconnect') 
            def on_disconnect(): 
                print("Disconnected from server") 
     
        def register_user(self, username): 
            """Register a new user with the server""" 
            response = requests.post( 
                f"{self.server_url}/api/users", 
                json={'username': username}, 
                headers={'Content-Type': 'application/json'} 
            ) 
            return response.json() 
     
        def get_user_info(self, username): 
            """Get information about a specific user""" 
            response = requests.get(f"{self.server_url}/api/users/amsharma") 
            return response.json() 
     
        def connect(self): 
            """Establish WebSocket connection with server""" 
            try: 
                self.sio.connect(self.server_url) 
                return True 
            except Exception as e: 
                print(f"Connection failed: {str(e)}") 
                return False 
     
        def send_message(self, message): 
            """Send a message to all connected clients""" 
            if self.sio.connected: 
                self.sio.emit('message', {'message': message}) 
                return True 
            return False 
     
        def disconnect(self): 
            """Disconnect from the server""" 
            if self.sio.connected: 
                self.sio.disconnect() 
     
    # Example usage 
    if __name__ == '__main__': 
        # Start server monitoring in background thread 
        monitor_thread = threading.Thread(target=monitor_connections, daemon=True) 
        monitor_thread.start() 
     
        # Start server 
        socketio.run(app, host='0.0.0.0', port=5000, debug=True)

This implementation demonstrates several key aspects of client-server architecture:

Server Components:

  • RESTful API endpoints for user management
  • WebSocket handlers for real-time communication
  • Connection monitoring and cleanup
  • Simulated database for data persistence

Client Components:

  • HTTP requests for user registration and information retrieval
  • WebSocket connection for real-time messaging
  • Event handlers for different types of server responses

Key Features:

  • Bidirectional communication using both HTTP and WebSocket protocols
  • Connection state management
  • Error handling and validation
  • Automatic cleanup of inactive connections
  • Broadcast messaging capability

Best Practices:

  • Separation of concerns between different types of communication
  • Proper error handling and status codes
  • Connection monitoring and resource cleanup
  • Event-driven architecture for real-time features

Understanding these architectural styles allows developers to choose the right approach based on system requirements, ensuring better quality, maintainability, and alignment with business objectives.
